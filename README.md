## 오프라인 배너 강조 · 개발자 옵션 핀 버튼화 · 픽업번호 버그 수정

1. **오프라인 배너 문구/색상**: "오프라인 — 최신 상태 아님" → "오프라인 상태 · 네트워크가
   연결되면 주문이 자동으로 반영돼요"로 문구를 다듬고(다른 안내 배너들과 같은 "상태 · 설명"
   톤에 맞춤), 배경색을 검정에서 포인트 레드로 바꿨다 — 야외 행사에서 오프라인은 치명적인
   문제라 확실히 눈에 띄어야 한다는 이유.
2. **개발자 옵션을 전역 핀 버튼으로 재구성**: 설정 리스트 안에 있던(→ 지난 단계에서 완전히
   제거됨) 개발자 옵션을, 화면이 무엇이든 항상 떠 있는 우하단 원형 핀 버튼(🛠️)으로 옮겼다.
   새 컴포넌트 `src/components/devOptionsPanel.js`가 `#app-root` 바깥의 전역 오버레이로
   렌더링되고, `router.js`가 화면을 바꿀 때마다 `DevOptionsPanel.refreshVisibility()`를
   호출해 "사장님(OWNER) 계정 + 매장 선택 완료" 상태일 때만 보이도록 한다(로그인 전/행사
   담당자 화면/로그아웃 후에는 숨김 — 실제로 로그인 화면·행사 담당자 홈에서 안 보이고,
   설정·메뉴관리·매출조회 등 어느 화면으로 이동해도 계속 떠 있고, 로그아웃하면 사라지는 것까지
   확인). 핀을 누르면 바텀시트가 열리고 기존 기능(오프라인 시뮬레이션 토글, 테스트 주문 생성)을
   그대로 담고 있다.
3. **'주문 넣기' 테스트 데이터 비율 조정**: `MockApi.createRandomOrder`의 `receiveType` 비율을
   50/50에서 픽업번호(COUNTER_PICKUP) 90% / 테이블번호(TABLE_SERVICE) 10%로 바꿨다(야외
   행사에서는 카운터 픽업이 훨씬 흔함). 테이블 번호는 기존처럼 1~20 사이 값을, 픽업번호는
   발권기 티켓처럼 4자리(1000~9999)로 생성하도록 자릿수도 함께 분리했다.
4. **픽업번호 검색 버그 발견 및 수정**: 점검 결과 검색 로직(`matchesSearch`, 부분 일치 —
   `order.tableOrPickupNo`에 대한 substring 매칭) 자체는 정확했다. 실제 버그는 카드에
   표시되는 값 쪽에 있었다 — `UI.pickupBlockHtml()`이 카운터 픽업 주문의 '픽업번호'로
   `order.tableOrPickupNo`(검색이 실제로 참조하는 필드)가 아니라 **고객 전화번호 뒷자리
   4개**를 대신 보여주고 있었다. 그 결과 화면에 보이는 픽업번호와 실제 검색 가능한 값이
   서로 달라서, 화면에 보이는 숫자를 그대로 검색하면 안 나오는 것처럼 보였다. 카운터
   픽업이든 테이블 서비스든 항상 `order.tableOrPickupNo`를 그대로 보여주도록 고쳤다 — 이제
   화면에 보이는 값과 검색되는 값이 항상 일치한다(정확히 일치/부분 일치/공백 포함 입력/
   존재하지 않는 값 등 여러 케이스로 재검증 완료).

## 메뉴 폼 필드 순서 · 설정 리스트 재배치 · 자동 품절 알림

1. **메뉴 등록/수정 폼 필드 순서 변경**: 메뉴명 → 카테고리 → 가격 → 설명 → 이미지 → 재고수량 →
   원산지 → 노출여부 → 옵션그룹 사용 → 미리보기 순으로 맞췄다(기존엔 재고수량/원산지가 설명/
   이미지보다 먼저 나왔음).
2. **설정 화면 리스트 재배치**: 영업상태 카드 → 메뉴 추가 및 수정 → 예상 대기시간 관리 →
   매출 조회 → 직원 계정 관리 → QR 메뉴판 보기 → 재고 소진 시 자동 품절 → 자동 수락 →
   매장 정보 → 로그아웃 순으로 맞췄다. '재고 소진 시 자동 품절'/'자동 수락'은 이제 독립된
   `.settings-list-item` 행(아이콘 + 라벨 + 토글)으로 승격되어 있고, 행 전체를 눌러도
   토글되도록 클릭 핸들러를 행에 붙였다(토글 자체만 눌러도 물론 동작). '개발자 옵션'
   (오프라인 시뮬레이션 토글·테스트 주문 생성 버튼)은 이 화면에서 완전히 뺐다 — 추후
   STEP 6에서 별도 위치로 옮길 예정이라 관련 CSS는 남겨두고 마크업/핸들러만 제거했다.
3. **메뉴 자동 품절 알림 추가**: 재고 수량을 0으로 저장해서 자동 품절이 발동되면
   (`MockApi.addMenuItem`/`updateMenuItem`이 `autoSoldoutTriggered` 플래그를 응답에 포함),
   화면 하단에 확인 버튼을 눌러야만 닫히는 알림을 띄운다(`UI.showPersistentBanner` — 토스트/
   배너와 달리 타이머로 자동으로 사라지지 않음, 새 host `#persistent-banner-host`). 여러
   메뉴가 한 번에 자동 품절되는 경우를 대비해 이름 배열을 받도록 만들었다(지금은 폼이 한 번에
   메뉴 하나만 저장해서 실제로는 항상 배열 길이 1).

## 예상 대기시간 관리 화면 개편 — 5단위 선택·최대값·미리보기(+추가 수정 반영)

1. **5단위 설정**: 대기 메뉴(N)/개당 예상 시간(M)/최대 예상 대기시간 값을 자유 입력이 아니라
   5단위로만 고를 수 있는 `<select>` 드롭다운으로 바꿨다(N: 5~50, M: 5~60, 최대값: 5~120).
   처음엔 −/+ 스테퍼 버튼으로 구현했다가, 사장님 피드백으로 "5단위로 고를 수 있는 선택 UI"로
   다시 바꿨다(`optionsHtml()` 헬퍼로 옵션 목록 생성).
2. **최대 예상 대기시간 추가**: `Store.waitTimeMaxMinutes`(기본 60분) 신설. 계산값이 이 값을
   넘으면 넘는 값 그대로가 아니라 이 값으로 고정한다(`MockApi.getEstimatedWaitInfo`가
   `rawEstimatedMinutes`(고정 전 원값)와 `estimatedMinutes`(고정 후 최종값)를 함께 반환).
3. **미리보기**: 한때 "①설정값 → ②가상 예시 → ③계산 과정 → ④고객 문구" 4단계로 풀어서
   보여주는 버전을 만들었는데, "이해하기 어렵다"는 사장님 피드백을 받아 단계별 UI는 삭제하고
   이전 화면으로 우선 원복했다 — 다만 그 아래 별도로 추가했던 '지금 실제 예상 대기시간'
   박스(현재 매장에 실제로 쌓여 있는 대기 메뉴 수량 기준 계산값, 최대값 고정 여부 표시 포함)는
   그대로 남겼다. 4단계 버전은 추후 다른 형태로 다시 다듬을 예정이라 완전히 폐기한 개념은
   아니다.
4. **토글 이름 변경 + 비활성화 연동**: '조리 예상 시간 안내 사용' → '예상 대기시간 안내 사용'으로
   이름을 바꿨다. 이 토글을 끄면 나머지 설정 요소(N/M·최대값 선택 드롭다운, 미리보기)를 감싸는
   `#wait-settings-body`에 `.wait-settings-disabled`(옅게 흐리고 클릭 자체가 안 먹도록
   `pointer-events:none`)를 적용해 전부 비활성화하고, 다시 켜면 그대로 복구된다.

## 주문 화면 — 메뉴 필터 바텀시트화 + 정렬 토글 추가

1. **필터 UI 변경**: 기존 '전체 메뉴' 드롭다운(`<select>`)을 필터 아이콘 + '필터' 라벨의 pill
   버튼으로 바꿨다. 누르면 바로 펼쳐지는 대신 화면 하단에서 올라오는 바텀시트
   (`.modal-overlay-bottom` — 메뉴 등록 폼에서만 쓰던 걸 두 번째 사용처로 확장)에서 메뉴를
   하나 골라 즉시 적용하고 닫힌다. 선택된 메뉴가 있으면 버튼 라벨이 '필터 · 바닐라라떼'처럼
   바뀐다. 대기/접수/완료 세 탭 모두 같은 함수(`renderMenuFilter`/`openMenuFilterSheet`)를
   그대로 재사용하며, 탭별 선택값은 기존처럼 독립적으로 유지된다(`menuFilterBySegment`).
2. **정렬 토글 추가**: 필터 버튼 옆에 정렬 아이콘 + '오름차순/내림차순' pill 버튼을 추가했다.
   누를 때마다 주문 시각 기준 오름차순↔내림차순이 토글된다(`sortDirection`, 세 탭 공통).
   시간대(5분 단위) 묶음 로직(`UI.groupByBucket`)은 입력이 시간순으로 정렬되어 있다고만
   가정하고 인접 항목의 버킷 경계만 비교하므로, 내림차순으로 넘겨도 최신 시간대가 먼저 나오는
   그룹 목록이 그대로 만들어진다 — 별도 처리 없이 자연스럽게 지원된다.

## 주문 화면 — 시간대 그룹 펼쳐보기/접기 재정의(카드 개별 접기 폐지)

시간대(5분 단위) 그룹의 '펼쳐보기'/'접기'가 그동안은 "그 시간대의 카드 목록을 아예 숨기거나
보여주는" 용도였는데, 이제는 "그 시간대 안의 모든 카드를 한 번에 상세 정보(PG 주문번호·주문
항목·결제금액·호출 이력)까지 펼치거나, 압축된 형태로 접는" 일괄 컨트롤로 바뀌었다. 카드
목록 자체는 이제 항상 보인다 — 예전처럼 그룹을 통째로 숨기지 않는다.

- 카드 하나하나에 따로 있던 '펼쳐보기/접기' 개별 버튼(카드 하단 링크 + 카드 본문 클릭)은
  삭제했다 — 이제 상세 표시 여부는 오직 소속된 시간대 그룹 단위로만 결정된다
  (`groupCollapsed[bucketKey]`를 각 카드 렌더링에 그대로 전달, `expandedIds` 같은 카드별
  상태는 더 이상 없다).
- PG 주문번호는 압축된 상태에서는 보이지 않고, 상세 정보 블록 맨 위(주문 항목 목록보다 먼저)에
  펼쳤을 때만 보인다.
- 화면 상단 '전체 접기/펼치기' 버튼은 코드 변경 없이 그대로 동작한다 — 원래도 같은
  `groupCollapsed` 상태를 모든 그룹에 한 번에 적용하는 방식이었어서, 상태의 '의미'만 바뀌고
  로직은 자연스럽게 새 정의를 따른다.
- 행사 담당자의 읽기 전용 '매장별 주문조회' 화면(eventManagerStoreOrders.js)은 애초에 이런
  펼치기/접기 개념이 없고(카드를 탭하면 모달로 상세를 보여주는 별개 구조) PG 주문번호도 항상
  보여주는 화면이라 그대로 두었다.

## 주문 화면 카드 표시 요소 정리 — 완료 횟수 표기 통일·되돌리기 버튼·pill 간격/크기

1. **완료 처리 버튼에 횟수 표기 통일**: 접수 탭의 '완료 처리' 버튼이 '고객 호출 (N회)'와 같은
   방식으로 항상 "완료 처리 (N회)"를 표시한다(0회부터). 되돌리기 후 다시 완료 처리하면
   숫자가 그대로 누적된다. 기존에 펼쳐보기(상세) 안에 따로 있던 "완료 처리 횟수: N회" 행은
   삭제했다(customers.js) — 버튼 라벨이 같은 정보를 대신 보여주므로 중복이었다. 정보를
   그대로 남겨야 하는 읽기 전용 화면(행사 담당자 매장별 주문조회, eventManagerStoreOrders.js)은
   완료 처리 버튼 자체가 없어 대체할 곳이 없으므로 그 화면의 상세 행은 그대로 유지했다.
2. **되돌리기 버튼 색상 수정**: 완료 탭의 '되돌리기'(`.btn-outline`)가 카드 배경(연회색/취소 시
   연분홍)과 거의 같은 색이라 안 보이던 문제 — 흰 배경 + 굵은 블랙 테두리로 바꿔 카드 배경과
   뚜렷이 구분되게 했다. 이 변경은 주문 카드 액션 버튼에만 스코프를 좁혀서
   (`.order-card-actions .btn-outline`) 모달/설정 화면 등 다른 곳의 `.btn-outline`(흰 배경
   위에서는 문제 없었음)에는 영향이 없다.
3. **pill 간격/크기 통일**: 채널 뱃지(QR오더/태블릿오더)와 주문유형 뱃지(포장/다회용기/체험)
   사이에 `margin-left`가 따로 붙어 있어 다른 요소 간 간격(8px)보다 넓게 벌어져 있던 걸
   제거했다. 높이도 이모지가 섞인 채널 뱃지가 패딩 계산만으로는 미묘하게 더 커서(26px vs
   22px) `height:26px`를 양쪽에 고정해 완전히 동일하게 맞췄다.

## 행사 담당자 '매출현황 및 분석' 화면 재구성 — 요약 확장·카드형 메뉴·AI 분석(목업)

1. **상단 요약 확장**: 기존 4개 지표(행사 전체 누적 매출/오늘 매출/참여 매장 수/매장당 평균
   매출)는 그대로 두고, 아래 2개를 추가했다.
   - **주문경로 비중(오늘)**: QR오더/태블릿오더 건수 비율을 %로 표시. 매출 기준인 '주문경로별
     매출' 도넛 차트와는 의도적으로 다른 지표다 — 이건 "몇 건이 어느 채널로 들어왔는지"를
     보여주는 건수 기준값이다(`MockApi.getEventOrderStats`).
   - **총 주문 / 취소(오늘)**: 완료 여부와 무관하게 오늘 접수된 모든 주문 건수와 그중 취소된
     건수.
2. **하단을 사장님 앱 '매출 조회'와 같은 카드형 방식으로 재구성**: 기존에 화면에 죽 나열되어
   있던 매장 랭킹/차트를 사장님 앱 매출 조회 화면과 동일한 허브(카드형 타일 + 3D 스타일
   이모지) → 상세(차트/랭킹) 구조로 바꿨다. 5개 카드: 매장별 매출(🏪, 랭킹), 결제수단별
   매출(💳, 도넛), 시간대별 매출(🕒, 바), 주문경로별 매출(🧾, 도넛), **메뉴별 매출(🍽️, 랭킹,
   신규)**. 메뉴별 매출은 행사에 참여한 모든 매장의 판매 데이터를 (매장, 메뉴명) 쌍으로 묶어
   집계해 "메뉴명 (매장명)" 형태로 보여준다(`MockApi.getEventMenuRanking`) — 매장이 다르면
   같은 메뉴명이 섞이지 않는다. 차트 SVG와 랭킹 목록(`UI.breakdownListHtml`, 이번에 sales.js
   에서 뽑아내 공유하도록 새로 추출)은 사장님 앱과 동일한 ui.js 공용 함수를 그대로 쓴다.
3. **AI 분석(목업)**: 'AI 분석' 버튼을 누르면 오늘/어제 매출 증감률, 오늘 매출 1위 매장,
   오늘 가장 많이 팔린 메뉴, 오늘 취소율을 계산해 미리 정해둔 문장 틀에 끼워 넣은 한 문단을
   보여준다(예: "오늘 매출은 어제보다 12% 증가했어요. OO부스가 가장 높은 매출을 기록했고...").
   **실제 AI/LLM 연동은 이번 단계 범위가 아니다 — `src/api/mockApi.js`의
   `getAiSalesInsight(eventId)` 함수 하나만 실제 API 호출로 교체하면 된다.** 반환 형태
   (`{ message: string }`)와 화면 쪽 호출부(`eventManagerSales.js`의 `toggle-ai-insight`
   버튼 핸들러)는 그대로 유지할 수 있도록 설계했다.

## 설정 화면 개편 — 예상 대기시간 자동 계산·직원 권한 관리·메뉴 순서·원산지·QR 메뉴판

'설정' 화면과 하위 화면들을 아래 7가지로 개편했다.

1. **매장 정보 위치 이동**: 목록 맨 위 근처에 있던 '매장 정보' 카드를 로그아웃 버튼 아래,
   개발자 옵션 바로 위(목록 맨 아래)로 옮겼다. 자주 안 바뀌는 정보라 자주 쓰는 설정 항목들보다
   아래로 내렸다.
2. **예상 대기시간 관리 화면 신설** (`waitTimeSettings.js`, 설정 > '예상 대기시간 관리'):
   기존엔 사장님이 '예상 대기시간(분)' 값 자체를 5분 단위로 직접 입력했는데(메뉴 관리 화면
   상단에 있었음), 주문이 쌓여도 자동으로 반영되지 않는 문제가 있었다. 이제는 "대기 메뉴
   (N)개당 예상 시간 (M)분" 두 기준값만 정해두면, 현재 대기·접수 탭에 남아있는 미완료 주문의
   메뉴 항목 총 수량을 기준으로 올림(총수량 ÷ N) × M 분으로 자동 계산한다
   (`MockApi.getEstimatedWaitInfo`). '조리 예상 시간 안내 사용' 토글을 끄면 고객 화면에 문구를
   노출하지 않는다. 설정값을 바꾸면 "고객에게 이렇게 보여요" 미리보기 문구가 실시간으로
   갱신된다. `Store.estimatedWaitMinutes` 고정값 필드는 제거하고
   `waitTimeGuideEnabled`/`waitTimeMenuCountUnit`/`waitTimeMinutesPerUnit`로 교체했다.
3. **직원 계정 관리 기능 강화** (`staffAccounts.js`, 신규 화면): 신규 직원 생성 폼(이름/로그인
   아이디/연락처/초기 비밀번호)과 직원별 권한 토글(주문 처리/메뉴 관리/매출 조회/설정 변경)을
   추가했다. 오너 계정은 항상 모든 권한을 가진 것으로 표시만 하고 이 화면에서 수정할 수
   없다. 비활성화된 직원 계정은 로그인 자체가 거부되도록 `MockApi.login`에도 검사를 추가했다.
4. **메뉴 노출 순서 변경**: '메뉴 추가 및 수정' 화면에서 같은 카테고리 안 메뉴끼리 위/아래
   이동 버튼으로 순서를 바꿀 수 있다 — 이미 있던 카테고리 순서 변경 기능과 동일한 방식
   (`MockApi.moveMenuItem`, sortOrder 맞바꾸기). '전체' 탭에서는 어떤 카테고리 기준으로
   순서를 매길지 애매해지므로 이동 버튼을 노출하지 않는다.
5. **원산지 표기**: 메뉴 등록/수정 폼에 '원산지' 입력 필드(선택, 빈 값 허용)를 추가하고,
   메뉴 카드와 고객 미리보기 카드에도 값이 있을 때만 노출한다.
6. **매출 계산 기준 통일**: 매출 집계 공통 전처리 함수(`ordersInRangeFor`)가 기존엔 취소된
   주문만 제외하고 대기·접수 중인 미완료 주문까지 매출에 포함하던 버그가 있었다 — 이제
   완료(COMPLETED) 상태인 주문만 포함하도록 고쳤다. 사장님 매출 조회·행사 담당자
   매출현황 분석 화면이 모두 이 함수를 공유하므로 한 곳만 고쳐서 양쪽에 반영된다. 사용되지
   않고 있던 `getDashboardSummary`의 매출 계산도 같은 기준으로 함께 고쳤다.
7. **QR 메뉴판 보기** (`qrMenu.js`, 신규 화면): 매장의 QR오더 메뉴판 링크(`AppConfig.QR_ORDER_BASE_URL`
   + storeId)와 그 링크를 인코딩한 QR코드 이미지를 보여주고 다운로드할 수 있다. 이 프로젝트는
   Node/npm 빌드 도구 없이 정적 파일로만 동작하기 때문에 QR코드 자체를 직접 그리는 라이브러리를
   번들링할 수 없어, 공개 QR 생성 API(api.qrserver.com) 이미지를 그대로 표시하는 방식을
   썼다 — 실제 서비스 전환 시에는 서버 측 QR 생성이나 정식 라이브러리로 교체를 권장한다.
   다운로드는 이미지를 fetch해 blob으로 받아 저장하며, 네트워크 문제로 실패하면 새 탭에서
   여는 방식으로 대체한다.

시드 데이터 스키마가 바뀌어 `DB_KEY`를 `v13`으로 올렸다(기존 로그인 세션과 함께 초기화된다).

## 주문 화면 — 채널 구분자·카드 상태 표시·동작 흐름 개편

'주문' 화면의 채널 배지 가독성, 카드 정보 체계, 발송 실패 처리, 그룹 펼침 UX, 완료 탭
동작, 자동/수동 수락, 메뉴별 필터까지 총 7가지를 개편했다.

1. **주문유형 pill 추가** (`Order.orderType`: `PACKAGING`(포장)|`REUSABLE`(다회용기)|`EXPERIENCE`(체험)).
   반영 전 실제 카드 배경(연회색 `--color-divider`) 위에 얹어 예시를 먼저 보여주고 승인받은
   뒤 반영했다 — 처음엔 파스텔 채움 배경을 제안했으나 카드 배경과 톤이 비슷해 잘 안 보인다는
   피드백을 받고, 흰 배경 + 글자와 동일한 색 아웃라인 스타일로 다시 디자인해 승인받았다.
   `UI.orderTypeBadgeHtml(order)`로 `ui.js`에 추가해 사장님 앱(`customers.js`)과 행사
   담당자 매장별 주문조회(`eventManagerStoreOrders.js`) 양쪽에서 재사용한다.
2. **QR오더 배지 색상 수정**: 기존엔 QR오더가 검정 채움 배경이라 🔳 이모지가 잘 안 보였다.
   QR오더는 흰 배경+검정 아웃라인, 태블릿오더는 검정 채움으로 스타일을 서로 바꿔 항상 잘
   보이게 했다(`channel-qr`/`channel-tablet` CSS).
3. **발송 실패 팝업**: 알림톡 발송이 실패하면 토스트 대신 "발송이 실패되었습니다" 중앙 팝업이
   뜨고 [다시 발송하기]/[닫기] 중 하나를 선택하게 했다(`doSendKakao`/`openSendFailureModal`).
4. **시간대 그룹 펼쳐보기/접기 라벨**: 기존 화살표 아이콘만 있던 그룹 헤더에 "펼쳐보기"/"접기"
   텍스트 라벨을 추가해 어떤 동작인지 명확히 했다.
5. **완료 탭 호출 버튼 제거**: 완료(DONE) 탭은 이미 조리·픽업이 끝난 주문이므로 고객 호출
   버튼을 없애고 [되돌리기]/[주문 취소]만 남겼다.
6. **자동/수동 수락 토글**: 설정 화면에 매장 단위 "자동 수락" 토글을 추가했다(`Store.autoAcceptOrders`,
   기본값 `false`=수동). 켜져 있으면 신규 주문이 대기(WAITING) 없이 바로 접수(RECEIVED)로
   들어온다(`createRandomOrder`가 이 값을 확인해 초기 상태를 정한다).
7. **메뉴별 필터**: 대기/접수/완료 3개 탭에 각각 독립적으로 동작하는 메뉴 필터 드롭다운을
   추가했다. 옵션은 전체 메뉴 목록이 아니라 실제 주문에 포함된 메뉴 이름만 나온다.

시드 데이터(`mockData.js`)의 `DB_KEY`를 `v12`로 올려 새 필드(`orderType`, `autoAcceptOrders`)가
포함된 데이터로 재시딩되게 했다(버전이 바뀌면 기존 `localStorage` 값을 버리고 새로 시딩하는
기존 규칙을 그대로 따름 — 로그인 세션도 함께 초기화된다).

## 주문 화면 — 상단 레이아웃 재배치 + 카드 정보 추가/수정

'주문' 화면 상단 구성과 카드 안 정보 표시 방식을 정리했다.

- 영업상태 표시를 상단바(topbar) 좌측으로 옮기고, 검색창은 토글 없이 항상 보이도록
  바꿨다(🔍 아이콘 + 픽업번호/테이블번호 검색). 계정 구분(관리자/직원 계정)과 "전체
  접기/펼치기" 버튼은 검색창 아래 별도 툴바 행으로 뺐다. '주문' 타이틀은 옆 요소들과
  무관하게 화면 중앙에 절대 위치로 고정해 항상 정중앙에 보이게 했다(`.topbar-title`).
- `Order.paymentMethod`(카드/간편결제/현금) 실제 필드를 추가해 금액 옆에 노출하고,
  매출 조회의 결제수단별 집계도 이제 이 필드를 직접 집계한다(예전엔 주문 id 해시로 흉내).
- 주문 시각을 24시간 형식으로 카드 상단에 강조 표시하고 "N분/시간 경과"를 함께 보여준다.
- 오입력 의심 전화번호 경고 문구를 "⚠️ 오입력 의심"으로 전화번호 옆에 짧게 붙였다(빨간
  강조는 유지).

## 행사 담당자 대시보드 — 실제 백엔드 연동 시 교체가 필요한 목업 API 목록

이 프로젝트 전체의 원칙대로, 아래 함수들은 전부 `src/api/mockApi.js` 안에 있고 **본문만**
`fetch(real endpoint)`로 바꾸면 화면 코드는 한 줄도 손댈 필요가 없다(함수 시그니처 유지).
행사 담당자 기능에서 새로 추가된 것만 정리하면:

| 함수 | 예정 엔드포인트 | 비고 |
|---|---|---|
| `getMyEvents(userId)` | `GET /users/:userId/events` | 담당 행사 목록 |
| `getEvent(eventId)` | `GET /events/:eventId` | 행사 상세 |
| `getStoresByEvent(eventId)` | `GET /events/:eventId/stores` | 행사에 속한 매장(부스) 목록 |
| `getEventDashboardSummary(eventId)` | `GET /events/:eventId/dashboard-summary` | 홈 대시보드 요약(매장 상태 집계/매출/주문건수) — **실서비스에서는 매장·주문 테이블에서 실시간 집계해야 함**, 지금은 `Store.todaySalesAmount` 등 미리 계산된 목업 필드를 그대로 합산 |
| `getAttentionStores(eventId)` | `GET /events/:eventId/attention-stores` | '주의가 필요한 매장' 감지 — 조건 A(대기 주문 지연)는 실제 Order 데이터 기반이라 그대로 이관 가능, 조건 B(`Store.lastOrderAt`)는 실서비스에서 주문 테이블 조인으로 대체 필요 |
| `bulkUpdateStoreStatus(storeIds, targetStatus, actorUser, scopeLabel)` | `POST /events/:eventId/stores/bulk-status` | 일괄 상태 변경 — **무작위 10% 실패 시뮬레이션은 목업 전용이므로 실서비스에서는 제거**하고 실제 매장 단말기/네트워크 응답에 따른 진짜 성공/실패로 교체 |
| `getAuditLogs(eventId)` | `GET /events/:eventId/audit-logs` | 감사 로그 조회 — 실서비스에서는 별도 감사 로그 테이블/서비스(예: 변경 불가능한 append-only 로그 저장소)를 쓰는 것을 권장 |
| `getEventSalesBreakdown(eventId, storeIdFilter, dimension, range)` | `GET /events/:eventId/sales` | 행사 전체/매장 필터 매출 집계 — `getSalesBreakdown`과 `computeBreakdown()`을 공유하므로 실서버 전환 시 두 엔드포인트를 같은 집계 쿼리(매장 범위만 다르게)로 구현하면 된다 |
| `updateStoreOperatingStatus(storeId, status)` | `PATCH /stores/:storeId/operating-status` | 기존 사장님 앱 함수를 행사 담당자의 개별 매장 변경에도 그대로 재사용 중 |

**목업이라 실제 데이터가 없는 부분(실서비스 전환 시 반드시 채워야 함)**:
- `Store.todaySalesAmount`/`totalSalesAmount`/`lastOrderAt`/`ownerName`/`ownerPhone` — store-1
  (브루웍스 성수점) 외 17개 부스는 실제 `Order`/`User` 데이터가 없어 표시용 목업 값을 직접
  넣어뒀다. 실서비스에서는 이 값들이 전부 실제 주문/계정 테이블에서 계산·조인된 값이어야 한다.
- '매장별 주문조회'의 미호출/호출 필터, '매장 사장님에게 연락하기' `tel:` 링크는 이미 있는
  실제 데이터(`NotificationLog`, `Store.ownerPhone`)를 그대로 사용하므로 구조 변경 없이 이관 가능.
- 감사 로그(`AuditLog`)는 지금 목업 DB의 배열 하나로 저장되지만, 실서비스에서는 위·변조 방지가
  중요한 데이터이므로 별도 저장소/보존 정책이 필요하다.

## 행사 담당자 매출현황 및 분석 화면

행사 담당자 셸의 '매출 현황' 탭이 실제 내용으로 채워졌다. `src/screens/eventManagerSales.js`
(신규 — 다른 탭 모듈들과 동일한 패턴)가 담당하며, **사장님 앱의 '매출 조회' 화면과 완전히 같은
차트 컴포넌트·색상**을 재사용한다(새로 만들지 않음).

**재사용을 위한 리팩터링**: `barChartSvg`/`donutChartSvg`/`donutLegendHtml`/`salesChartHtml`
(막대·도넛 차트 SVG 생성 + 카드 조립 로직)을 `src/screens/sales.js`에서 `src/components/ui.js`로
옮겼다 — 이제 `UI.salesChartHtml(dimension, data)` 하나로 사장님 앱과 행사 담당자 앱이 완전히
같은 코드를 호출한다(복붙이 아니라 진짜 공유). 데이터 집계 쪽도 마찬가지로 `mockApi.js`에서
`computeBreakdown(ordersInRange, dimension)`을 추출해서, 매장 하나만 보는 기존
`getSalesBreakdown(storeId, ...)`과 행사 전체/매장 필터를 지원하는 신규
`getEventSalesBreakdown(eventId, storeIdFilter, dimension, range)`이 같은 집계 로직을 공유한다.

**화면 구성**
1. 상단 요약: 행사 전체 누적 매출, 오늘 매출, 참여 매장 수, 매장당 평균 매출(누적 매출 ÷
   매장 수) — `MockApi.getEventDashboardSummary(eventId)`를 그대로 재사용(홈 대시보드와 동일 API).
2. 기간 필터: 오늘 / 행사 전체 기간(`Event.startDate`~`endDate`) / 직접 선택(커스텀 날짜) —
   사장님 앱 `sales.js`의 `date-range-bar`/`date-range-custom` 컴포넌트 그대로 재사용.
3. **매장별 매출 랭킹**: 매출 높은 순 정렬 + 최댓값 대비 상대 비율만큼 채워지는 막대(1등만
   포인트 컬러인 검정, 나머지는 회색 — 막대/도넛 차트와 같은 "최댓값만 강조" 색상 규칙을 그대로
   따름). 기간이 '오늘'이면 `Store.todaySalesAmount` 기준, 그 외(전체 기간/커스텀)에는
   `Store.totalSalesAmount`(누적) 기준으로 정렬한다 — 대부분의 부스가 날짜별 실제 주문 데이터를
   갖고 있지 않아(행사 담당자 데이터 구조 개편 단계 참고) 기간별로 정교하게 나눌 수 없는 대신,
   솔직하게 "오늘 vs 누적" 두 가지 기준만 제공한 것. 항목을 누르면 '매장별 주문조회' 자리로
   이동한다(이 모듈 내부의 플레이스홀더 뷰로 전환 — 아직 뒤로가기만 가능, STEP 5에서 실제 내용
   구현 예정).
4. 시간대별 매출 추이(막대), 5. 결제수단별 비중(도넛), 6. 주문경로별 비중(도넛) — 전부
   `UI.salesChartHtml()` 재사용, 데이터는 `MockApi.getEventSalesBreakdown()`에서 가져온다.
5. **매장 필터**(드롭다운, 기본값 '전체 합산'): 4~6번 차트에만 적용된다(매장별 랭킹은 정의상
   교차비교라 필터 대상에서 제외). store-1(브루웍스 성수점) 외에는 실제 주문 데이터가 없으므로,
   다른 매장을 단독으로 선택하면 정직하게 "해당 기간의 매출이 없어요"로 표시된다(데이터를
   지어내지 않음 — 실제로 확인함).

**변경/신규 파일**: `src/components/ui.js`(차트 헬퍼 이전 + `salesChartHtml` 추가),
`src/screens/sales.js`(로컬 차트 함수 제거하고 `UI.*` 호출로 교체 — 화면 동작은 동일),

**일괄 제어(전체·선택 공통 로직)**: 화면 상단 '전체 개점/일시중지/마감' 버튼과, 체크박스로
매장을 선택하면 하단에 뜨는(`position: sticky` 고정) 액션바의 '선택 매장 개점/일시중지/마감'
버튼이 **완전히 같은 함수**(`MockApi.bulkUpdateStoreStatus` + `eventManagerStores.js`의
`runBulkAction`)를 탄다.
1. 확인 팝업(화면 중앙)에서 실제 영향받는 매장 수를 미리 계산해 보여준다 — 예: "전체 12개
   매장을 개점할까요? 이미 영업중 상태인 매장 7개는 제외됩니다."
2. 확인을 누르면 대상 매장을 순회하며 상태를 바꾼다. 이미 같은 상태인 매장은 건드리지 않고
   자동 제외(skipped)하며, 나머지는 실제로 바꾸되 **약 10% 확률로 무작위 실패를 재현**한다
   (테스트를 위해 일부러 섞은 실패 케이스 — 매장 단말기 오프라인 등을 흉내). 하나가 실패해도
   나머지 매장 처리는 계속 진행한다(부분 실패 허용).
3. 처리 결과를 요약 팝업으로 보여준다: "성공 N개 · 이미 같은 상태라 제외 N개 · 실패 N개" +
   실패한 매장명 목록. 실패가 있으면 **'실패 건 재시도'** 버튼이 떠서, 실패했던 매장만 다시
   `bulkUpdateStoreStatus`를 호출한다(재시도도 같은 10% 확률 실패를 다시 탈 수 있어 여러 번
   눌러야 전부 성공할 수도 있다 — 실제 테스트에서 확인함).
4. 처리 1건마다 `AuditLog`에 한 건씩 기록한다: `"행사담당자 {이름}님이 {전체 N개|선택한 N개}
   매장을 {상태}(으)로 변경"` + `resultSummary`("성공 N개 · 이미 같은 상태라 제외 N개 · 실패
   N개") + `timestamp`. 화면 하단 **'최근 조치 이력'** 섹션에서 최신순으로 확인할 수 있고,
   `console.log('[AuditLog]', ...)`로도 함께 출력된다.

**개별 매장 상태 변경**: 리스트의 각 매장 카드에는 현재 상태에서 갈 수 있는 다음 상태 버튼이
바로 붙어 있다(사장님 앱 설정 화면의 `actionsFor(status)`와 같은 패턴 — OPEN이면 '일시중지'/'마감',
PAUSED면 '개점'/'마감', CLOSED면 '개점'만). 이건 일괄 제어와 의도적으로 다르게 설계했다 —
"바로 바꿀 수 있는" 가벼운 조작이라는 요구사항에 맞춰 **확인 팝업 없이 즉시 반영**되고, 실패
시뮬레이션도 없고, AuditLog에도 남기지 않는다(감사 로그는 일괄 조치 전용).

**홈 대시보드 연결**: STEP 2에서 만든 홈 대시보드의 '전체 매장 마감/개점' 빠른 액션 버튼이

## 행사 담당자 홈 대시보드

행사 담당자 셸의 '홈' 탭이 실제 내용으로 채워졌다. "① 행사가 잘 돌아가고 있는가, ② 지금
개입해야 할 곳이 있는가, ③ 매출은 어느 정도인가"에 즉시 답할 수 있는 정보만 담고, 개별
매장의 메뉴/재고 같은 세부 운영 정보는 다루지 않는다(설계 원칙 그대로).

**화면 구성** (`src/screens/eventManagerHome.js`, 신규 — Router 화면이 아니라 `eventManagerShell.js`의
'홈' 탭이 직접 render/mount/unmount를 호출하는 하위 모듈)
1. 상단: 행사명, 기간("2026.07.04 ~ 07.09" 형태), 며칠째인지("3일차"), 로그인한 담당자 이름.
2. 매장 상태 요약 카드: 영업중/일시중지/마감 매장 수 + 총 매장 수(색상은 기존 `.operating-status-dot`
   점 재사용 — 새 배지 색을 만들지 않았다). 누르면 '매장 현황' 탭으로 이동(`em:goto-tab` 커스텀
   이벤트로 셸에게 탭 전환을 요청 — 홈 모듈은 셸의 탭 상태를 직접 건드리지 않는다).

4. 오늘 총 주문건수 카드.
5. **'주의가 필요한 매장'** — 아래 3가지 조건을 매번 실제 데이터로 감지한다(하드코딩된 목록이 아님):
   - **A. 대기 지연**: WAITING 주문이 15분 이상 접수되지 않은 매장 — 실제 `Order.orderedAt` 기준으로 계산.
   - **B. 신규 주문 없음**: 영업중인데 `Store.lastOrderAt`이 1시간 이상 전인 매장.
   - **C. 장시간 마감**: 행사가 '진행중'인데 `Store.statusChangedAt` 기준 마감 상태를 30분 이상
     유지 중인 매장(행사가 '예정'이라 아직 시작 전인 매장의 마감은 정상 상태라 제외했다).
   `MockApi.getAttentionStores(eventId)`가 매번 새로 계산하며, 시드 데이터에 세 조건을 각각
   최소 1개씩 트리거하도록 픽스처를 넣어뒀다: 조건 A는 타코야끼 부스(store-3)에 25분 전 WAITING
   주문, 조건 B는 크래프트비어 하우스(store-4)를 영업중 + 95분 전 마지막 주문으로, 조건 C는
   기존 시드에서 이미 CLOSED로 배정된 떡볶이 포차(store-5)/핫도그 트럭(store-9)이 자연스럽게
   해당된다. 실제 확인해보니 store-1(브루웍스 성수점)의 실제 주문(order-5, 15분 전 WAITING)도
   조건 A에 우연히 걸려 함께 노출됨 — 하드코딩이 아니라 진짜 계산 로직이라는 걸 보여주는 사례.
6. 빠른 액션 버튼 '전체 매장 마감' / '전체 매장 개점' — 누르면 화면 중앙 확인 팝업만 뜨고,

## 주문 화면 — 상태 구조 재개편(대기 → 접수 → 완료)

- 상태 탭을 **대기(WAITING) / 접수(RECEIVED) / 완료(COMPLETED+CANCELED)** 3개로 재구성했다.
  기존 '호출' 탭은 삭제했다 — 호출은 이제 접수 탭 안에서 반복 가능한 액션일 뿐 별도 상태가 아니다.
  QR/태블릿 신규 주문은 항상 **대기** 탭에 먼저 들어가고, 대기 탭 카드의 **'주문 수락'** 버튼을
  눌러야 접수 탭으로 넘어간다. 접수 탭에서는 '고객 호출'을 몇 번이든 누를 수 있고(호출 시마다
  버튼 옆 "(N회)"가 올라간다), '완료 처리'를 누르면 완료 탭으로 넘어간다. 완료 탭의 '되돌리기'는
  다시 접수 탭으로 되돌린다(더 이상 돌아갈 '호출' 탭이 없으므로).
- **완료 처리 누적 횟수**(`Order.completedCount`)를 새로 추가했다 — COMPLETED로 전이될 때마다 1
  증가하고, 되돌리기는 이 값을 줄이지 않는다(되돌렸다가 다시 완료 처리하면 2, 3...으로 계속
  누적). 카드 앞면에는 노출하지 않고 '펼쳐보기' 상세 안에 "완료 처리 횟수" 항목으로만 보여준다.
  `src/api/mockApi.js`의 `advanceOrderStatus`/`revertCompletedToReceived`(기존 `revertCompletedToReady`에서
  개명)에서 관리하며, localStorage 목업 DB(`sajang-app-mock-db-v7`)에 그대로 저장된다.
- 카드 안 카카오 알림톡 **미리보기 문구와 '성공' 표시를 삭제**했다 — 호출 시 문구/버튼 없이 바로
  발송 처리되고 호출 횟수만 올라간다. 발송 **실패** 안내와 재발송(같은 '고객 호출' 버튼 재클릭)
  옵션은 그대로 유지했다.
- 채널 뱃지에 이모지를 추가했다: 태블릿오더 🖥️, QR오더 🔳.
- 오입력 의심 전화번호를 자릿수와 무관하게 항상 `000-0000-0000` 형태(3-4-4 하이픈)로
  포맷팅해서 보여주도록 `formatFullPhone`을 수정했다(기존엔 11자리가 아니면 하이픈 없이 그대로
  나열했음). 빨간 볼드 + "오입력 가능성 있음" 문구는 그대로 유지.
- 화면 상단 검색 아이콘을 누르면 픽업번호/테이블번호로 현재 탭의 주문을 필터링하는 검색창이
  나타난다(`tableOrPickupNo` 필드 부분일치 검색).

## 메뉴 관리 화면 개편

- **'전체' 카테고리 탭 추가**: 카테고리 탭 맨 앞에 '전체'가 추가되어 카테고리 구분 없이 모든
  메뉴를 한 리스트로 볼 수 있다(기본 진입 화면도 이제 '전체'). '전체' 보기에서는 각 메뉴 카드에
  소속 카테고리 배지를 함께 표시해 어떤 카테고리인지 구분할 수 있게 했다.
- **품절 토글 라벨**: 메뉴 리스트의 품절 토글 위에 작은 '품절' 캡션을 항상 표시해 토글의 용도를
  명확히 했다.
- **노출 여부 안내 문구**: '노출 여부' 토글 아래에 상태에 따라 문구가 바뀌는 안내를 추가했다 —
  켜짐: "지금은 고객님 화면에 잘 보이고 있어요." / 꺼짐: "꺼두면 고객님 화면에서 잠시 숨겨져요.
  나중에 다시 켜면 바로 보여드릴 수 있어요."(해요체·긍정적 톤).
- **재고 관리**: 메뉴별 '재고 수량' 입력(선택, 0 이상 숫자, 비워두면 무제한)과 매장 단위 '재고
  소진 시 자동 품절' 설정(설정 화면, 기본값 켜짐)을 추가했다. 재고를 0으로 저장하는 순간
  자동 품절 설정이 켜져 있으면 즉시 `isSoldout=true`로 전환된다(꺼져 있으면 전환 안 함 — 두
  경우 모두 직접 테스트로 확인). 메뉴 리스트에는 재고가 설정된 메뉴에 한해 "재고 N개"를 작게
  표시한다.
- **예상 조리 시간**: 메뉴별 '예상 조리 시간(분)' 입력을 추가하고, 메뉴 리스트 카드와 등록/수정
  폼의 고객 미리보기 카드 양쪽에 "예상 조리시간 O분"으로 표시한다.
- **옵션별 품절 처리**: 옵션 그룹 안 각 옵션 항목에도 개별 품절 토글을 추가했다. 품절 처리된
  옵션은 고객 미리보기 카드에서 취소선 + "· 품절" 표시로 선택 불가임을 나타낸다.
- **카테고리 관리 위치**: 카테고리 추가/수정/숨김/삭제/순서 변경 기능은 원래부터 설정 화면이
  아니라 메뉴 관리 화면 자체의 상단바(우측 아이콘)에 있었고, 이번에도 그대로 유지했다 — 설정
  화면 목록에는 카테고리 관리 항목이 없다.

  빨간 배경+테두리로 구분한다(기존에 있던 "신규 주문 빨간 테두리" 강조는 제거 — 이제 빨간색은
  취소된 주문에만 쓴다).
- **액션 버튼 색상 구분**: 고객 호출=블랙(`btn-primary`), 완료 처리=그린(`btn-success`),

  화이트+블랙 테두리 필로 구분하고 볼드 처리했다.
- **카드 레이아웃 변경**: 좌측에 대표 메뉴명("바닐라라떼" 등, 크고 굵게), 우측 상단에
  픽업번호/테이블번호(라벨은 작게, 값은 크고 굵게) — 요청 예시(픽업번호 6666 + 바닐라라떼)와
  동일하게 배치. 결제금액은 버튼이 아닌 일반 텍스트로 변경.
- **전화번호 마스킹 해제 + 오입력 경고**: 전체 번호를 그대로 표시하고, 010으로 시작하지 않거나
  11자리가 아니면 ⚠️ + 빨간 굵은 글씨 + "오입력 가능성 있음" 보조 문구를 노출한다. 시드 데이터에
  테스트용 오입력 번호 2건을 심어뒀고, 실시간 시뮬레이터도 약 15% 확률로 오입력 번호를 생성한다.
- **시간대 그룹 접기/펼치기 + 일괄 선택**: 각 5분 그룹을 개별로 접고 펼 수 있고, 상단
  '전체 펼치기/접기'로 전체를 한 번에 제어한다. 그룹 헤더의 체크박스로 그룹 내 전체 선택도
  가능하고, 1건 이상 선택 시 그룹 상단에 일괄 액션바("N건 선택됨" + 일괄 호출 + 일괄 완료처리,
  완료 탭에서는 일괄 완료처리 버튼 숨김)가 나타난다.
- **호출 관련 정책 변경**: '카톡으로 호출' → **'고객 호출'**로 이름 변경. 재호출 전 확인 팝업과
  '다시 보내기' 버튼을 없애고, 접수/호출 탭 어디서든 같은 '고객 호출' 버튼을 눌러 바로 발송하며
  호출 시도 횟수(`NotificationLog` 건수)를 버튼 옆에 "(N회)"로 누적 표시한다. 카드에 있던
  '호출완료'/'발송 실패' 상태 배지는 삭제했다(호출 이력은 펼쳐보기 안에서 계속 확인 가능).
- **완료 탭 안전장치**: 완료 탭에는 '완료 처리' 버튼 대신 **'되돌리기'** 버튼을 두어 실수로 완료
  처리한 주문을 READY(호출) 상태로 되돌릴 수 있다(`MockApi.revertCompletedToReady` 신규 — 유일하게
  역방향 상태 전이를 허용하는 예외 함수).

### ⚠️ 정책 완화 — "접수 단계에서도 고객 호출 항상 가능"

기존 STEP3 규칙은 "READY(준비완료) 상태에서만 호출 가능"이었는데, 이번 요청으로 **주문 상태와
무관하게(취소 제외) 항상 호출 가능**하도록 의도적으로 완화했다. 영향을 주는 지점:

- `MockApi.buildKakaoMessage`가 반환하는 고정 문구는 여전히 "주문하신 메뉴가 **준비되었습니다**"
  이다. 이제 접수(RECEIVED) 단계에서도 호출이 가능해졌으므로, 아직 준비되지 않은 주문에 대해
  호출하면 실제 상태와 문구가 어긋날 수 있다. 이번 작업 범위에는 문구 수정이 포함되지 않아 그대로
  뒀다 — 문구를 상태별로 분기하려면 `buildKakaoMessage`와 `sendKakaoAlert`를 함께 손봐야 한다.
- `MockApi.sendKakaoAlert` 자체는 원래도 상태 검증 없이 동작했기 때문에(주문 존재 여부만 확인)
  API 레벨 변경은 없었다 — 이번 변경은 전적으로 화면(customers.js)의 버튼 `disabled` 조건 완화다.

## ⚠️ 이번 단계에서 스택을 바꾼 이유

원래 계획은 **Expo + React Native + TypeScript** 네이티브 앱이었지만, 이 작업을 실행한
컴퓨터에 **Node.js가 설치되어 있지 않아** Expo 프로젝트를 생성/실행/검증할 수 없었습니다.
사용자 확인 결과 이번 결과물은 "개발자에게 보여줄 목업" 용도이므로, **순수 HTML/CSS/
바닐라 JS**로 동일한 화면 구조·디자인 시스템·데이터 모델·API 계층 분리를 그대로 재현했습니다.

- Node.js/npm이 없어도 브라우저에서 바로 열립니다 (빌드 단계 없음).
- 폴더 구조, 컴포넌트 이름, API 함수 시그니처, 데이터 모델은 **실제 Expo 프로젝트로
  옮길 때 거의 그대로 재사용**할 수 있도록 설계했습니다. (아래 "실제 Expo 프로젝트 전환 방법" 참고)
- Node.js가 설치되면 이 문서의 안내에 따라 실제 RN 프로젝트로 옮기는 걸 권장합니다.

## 실행 방법

Node.js 없이 정적 파일만으로 동작하므로, 아래 둘 중 하나로 열면 됩니다.

1. **가장 간단히**: `index.html` 파일을 더블클릭해서 브라우저로 바로 엽니다.
2. **로컬 서버로(권장, 새로고침 시 상태 유지 등 실제 앱과 더 비슷하게 동작)**:
   - PowerShell에서 `server.ps1`을 실행합니다.
     ```powershell
     powershell -NoProfile -ExecutionPolicy Bypass -File .\server.ps1 -Port 8935
     ```
   - 브라우저에서 `http://localhost:8935` 접속.

데모 로그인 계정: **아이디 `owner` / 비밀번호 `1234`** (로그인 화면에 미리 채워져 있음)

## 확인된 동작 (완료 조건)

- 로그인 → 홈 화면 → 하단 탭(홈/주문/고객/메뉴) 전환 → 설정 → 로그아웃까지 정상 동작 확인.
- 매장이 1곳만 시드되어 있어 로그인 직후 매장 선택 화면은 자동 스킵되고 바로 홈으로 이동.
  (매장 선택 화면 로직 자체를 보고 싶다면 `index.html?forceStoreSelect=1` 로 접속)
- 홈 화면 영업상태 토글 ON/OFF, 오늘 매출/주문건수/대기 고객 수 요약 카드가 목업 주문
  데이터로부터 계산되어 표시됨.
- 디자인 시스템(블랙앤화이트 + 예외적 포인트 레드)이 로그인/홈/설정/탭바 전반에 적용됨.

## 디자인 시스템 (공용 컴포넌트)

`src/components/ui.js` + `src/styles/components.css` 에 정의되어 있고, 모든 화면이 이걸
재사용합니다. 다음 단계(주문 수신함 등)에서도 반드시 이 컴포넌트를 재사용해야 합니다.

| 컴포넌트 | 함수/클래스 | 비고 |
|---|---|---|
| Button | `UI.button({label, action, variant})` / `.btn-primary` 등 | variant: primary/secondary/outline/text/danger-text |
| Card | `.card`, `.card-list-item`, `.summary-card` | 대시보드 요약 카드 포함 |
| Badge | `UI.badge(label, variant)` | variant: neutral/dark/danger/danger-soft — **danger 계열은 신규/품절/취소/오류에만 사용** |
| Toggle | `UI.toggle(isOn, action)` / `.toggle` | 영업상태 ON/OFF 등 |
| TabBar(하단) | `UI.tabBar(activeTab)` / `.tabbar` | 홈/주문/고객/메뉴 |
| SegmentTab(상단, 다음 단계용) | `.segment-tabs`, `.segment-tab` | 주문 탭의 대기/호출/완료 등 상태 탭에서 재사용 예정 |
| TopBar | `UI.topBar({title, leftIcon, leftAction, rightIcon, rightAction})` | 좌상단 설정 아이콘, 뒤로가기 등 |
| Toast | `UI.showToast(message)` | 하단 CTA 위에 잠깐 표시 |

컬러 토큰(`src/styles/tokens.css`): `--color-text-primary(#111111)`, `--color-bg(#fff)`,
`--color-divider(#F2F3F5)`, `--color-disabled(#E5E8EB)`, `--color-text-secondary(#8B95A1)`,
`--color-accent-red(#E5484D, 신규/품절/취소/오류 전용)`. 폰트는 Pretendard를 우선 지정하고
시스템 폰트로 폴백(`--font-family-base`) — 실제 앱에서는 `expo-font`로 Pretendard 폰트
파일을 번들에 포함시키면 됩니다.

## 목업 API 엔드포인트 (`src/api/mockApi.js`)

화면 코드는 아래 함수만 호출합니다. 실제 백엔드가 정해지면 **함수 시그니처는 그대로 두고
함수 본문만 fetch 호출로 교체**하면 화면 코드는 손댈 필요가 없습니다.

| 함수 | 예정 REST 엔드포인트 |
|---|---|
| `MockApi.login(loginId, password)` | `POST /auth/login` |
| `MockApi.getMyStores(userId)` | `GET /users/:userId/stores` |
| `MockApi.getStore(storeId)` | `GET /stores/:storeId` |
| `MockApi.updateStoreOperatingStatus(storeId, status)` | `PATCH /stores/:storeId/operating-status` |
| `MockApi.getDashboardSummary(storeId)` | `GET /stores/:storeId/dashboard-summary` |
| `MockApi.getMenuCategories(storeId)` | `GET /stores/:storeId/menu-categories` |
| `MockApi.getMenuItems(storeId)` | `GET /stores/:storeId/menu-items` |
| `MockApi.getOrders(storeId)` | `GET /stores/:storeId/orders` |
| `MockApi.getNotificationLogs(storeId)` | `GET /stores/:storeId/notification-logs` |
| `MockApi.resetSeed()` | (개발용) 시드 데이터로 초기화 |

목업 DB는 `localStorage`(`sajang-app-mock-db-v1`)에 저장되어, 새로고침해도 영업상태
토글 등 변경 사항이 유지됩니다. 완전히 초기 상태로 되돌리려면 브라우저 개발자도구
콘솔에서 `localStorage.clear()` 후 새로고침하세요.

## 실제 Expo(React Native + TypeScript) 프로젝트로 전환하는 방법

Node.js 설치 후, 아래 순서로 이 목업을 실제 네이티브 프로젝트로 옮기는 것을 권장합니다.

1. **프로젝트 생성**
   ```bash
   npx create-expo-app@latest sajangnim-admin-app -t expo-template-blank-typescript
   cd sajangnim-admin-app
   npm install zustand @tanstack/react-query @react-navigation/native @react-navigation/native-stack @react-navigation/bottom-tabs
   npx expo install react-native-screens react-native-safe-area-context expo-font
   ```
2. **상태관리 선택 이유**
   - **Zustand**: 로그인 유저/선택 매장/영업상태처럼 앱 전역에서 읽고 쓰는 가벼운 상태에 적합 (이 목업의 `src/state/store.js`가 Zustand `create()` 시그니처로 바로 옮길 수 있게 설계되어 있음).
   - **React Query**: 주문/메뉴 목록처럼 서버(목업 API 포함) 데이터를 가져오고 캐싱·리프레시해야 하는 부분에 적합. `src/api/mockApi.js`의 각 함수를 그대로 `queryFn`으로 감싸면 됨.
3. **폴더 구조 매핑**
   - `src/screens/*.js` → `features/{auth,home,orders,customers,menu,settings}/screens/*.tsx`
   - `src/components/ui.js` → `shared/components/{Button,Card,Badge,Toggle,TabBar,TopBar,Toast}.tsx`
   - `src/api/mockApi.js` → `shared/api/mockApi.ts` (json-server로 교체 시 `shared/api/client.ts`에서 baseURL만 바꾸는 구조 권장)
   - `src/data/mockData.js` → `shared/api/mock/seed.json` (json-server의 `db.json`으로 사용 가능)
   - `src/styles/tokens.css` → `shared/theme/tokens.ts`
4. **목업 API 서버**: Node 사용 가능 환경에서는 `json-server`로 교체 권장.
   ```bash
   npm install -D json-server
   npx json-server --watch shared/api/mock/seed.json --port 4000
   ```
   이후 `mockApi.ts` 내부를 `fetch('http://localhost:4000/...')`로 교체.
5. **네비게이션**: `router.js`(초경량 라우터) → `@react-navigation`의 `NativeStackNavigator`(로그인/매장선택/설정) +
   `BottomTabNavigator`(홈/주문/고객/메뉴)로 교체.
6. **폰트**: Pretendard 폰트 파일(otf/ttf)을 `assets/fonts/`에 넣고 `expo-font`의 `useFonts`로 로드.

## 기획 주석 보드 (desc/) — 화면별 번호 핀 + 3분류 설명 카드 + 메모

앱 전체 21개 화면(사장님 로그인~설정, 행사 담당자 로그인~매장별 주문조회)을 실제 캡처 화면 위에 번호
핀을 찍고, 번호별로 **기능·목적 / 동작·상태값 / 예외처리·UX문구** 3분류 카드로 정리한 정적 문서를
`desc/` 폴더에 추가했다. 기능명세서 작성(Claude Cowork 등)이나 Figma 임포트, 팀 리뷰용 근거 자료로
쓰기 위한 것으로, 실제 앱 동작에는 전혀 관여하지 않는다.

- **보러 가기**: [desc/description.html](desc/description.html) (전체 화면 한 장) 또는
  [desc/boards/index.html](desc/boards/index.html) (화면별로 분리된 개별 페이지 — Figma로 가져갈 때는
  이쪽을 화면 단위로 임포트하는 것을 권장, 레이어가 기능별로 깔끔하게 나뉜다).
- **카드 아래 "내 메모" 칸**: 각 번호 카드에 바로 메모를 적을 수 있고, 입력한 내용은 그 브라우저의
  `localStorage`에 자동 저장된다(새로고침해도 유지) — 요청하신 "기능마다 번호를 넣고 메모할 수 있게"를
  이 방식으로 구현했다. 다만 다른 기기·브라우저에는 동기화되지 않으니, 확정된 내용은 별도 문서로 옮겨두는
  게 안전하다.
- **구조**:
  - `desc/screens.js` — 화면 순서·이름·그룹·핀 좌표(퍼센트) 데이터.
  - `desc/details.js` — 핀 번호별 3분류 설명 데이터. **내용만 고칠 때는 이 파일만 수정하면 되고, 이미지는
    다시 만들 필요가 없다** (화면에서 `render.js`가 두 데이터를 합쳐 그린다).
  - `desc/img/<key>.png` / `<key>_pin.png` — 화면 원본 캡처 / 번호 핀 합성본.
  - `desc/board.css`, `desc/render.js` — 공용 스타일·렌더 로직.
  - `desc/REVIEW_NOTES.md` — 작성 중 발견한, 제품 의도 확인이 필요한 항목 메모(가로 스크롤이 필요한 탭
    구성, "완료 처리 (0회)" 표기 등 3건).
- **캡처 방식**: `index.html` 끝에 `?cap=<key>`일 때만 동작하는 비침투 진입점을 추가해, 평소 앱 동작에는
  전혀 영향이 없다. 로그인·매장/행사 선택·탭 전환을 코드로 대신 밟은 뒤 화면을 실제 콘텐츠 높이만큼 펼쳐
  캡처했다.
